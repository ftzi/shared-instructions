# i18n Best Practices

Internationalization rules for any project with locale files. Applies to all apps (mobile, web, landing pages) and any locale count (2 or 27).

## Source of Truth

- The source locale file (e.g. `en.json`) is the canonical voice. Translations MUST match its intent, tone, and structure in their own language, not just word-substitute.
- Use native words, not foreign loanwords, when a native equivalent exists. Don't use "tiers" or "upsell" in Portuguese when "niveis" and "venda adicional" exist. Common tech terms accepted in the target language (e.g. "web" in Portuguese) are fine.
- If a meaning changes, the source locale MUST change first, then translations follow. NEVER let a translation diverge from the source's intent.

## All User-Facing Text MUST Be i18n

NEVER assemble user-facing strings in const variables or template literals in source code.

Bad:
```ts
const billedAmount = `$${YEARLY_PRICE}/year`;
const trialPrice = yearly ? `$${YEARLY_PRICE}/yr` : `$${MONTHLY_PRICE}/mo`;
```

This bakes English words, currency symbols, units, and period suffixes (`/year`, `/mo`, `/yr`) into the code, making them impossible to translate.

Good:
```ts
const billedPrice = YEARLY_PRICE;
t("pricing.billed", { amount: billedPrice });
// en.json: "Billed at ${{amount}}/year"
// pt.json: "Cobrado ${{amount}}/ano"
```

Any text a user reads, including interpolated currency, units, and suffixes, MUST live in a locale file and be rendered through `t()`. Pass raw values (numbers, dates) as interpolation params. The translation owns every word, symbol, and unit around them.

## NEVER Put Code Identifiers in i18n Files

Locale files MUST contain only translatable text. NEVER put component names, icon names, route paths, CSS classes, or any other code identifier in a locale file.

Bad:
```json
"apps": [
  { "name": "Todo app", "icon": "ListChecks" },
  { "name": "Habit tracker", "icon": "Calendar" }
]
```

The `"icon": "ListChecks"` value is identical in every language, inflates every locale file, and drifts from the code it references.

Good: keep only the label in the locale file. Define the code mapping in the component by index or a parallel constant:
```ts
const appIcons = [ListChecks, Calendar, ...];
const apps = objectValues<string>(t("oneapp.apps", { returnObjects: true }));
```

## No Arrays in i18n Files

NEVER use JSON arrays in locale files. Use objects with `item0`, `item1`, ... keys instead.

Bad:
```json
"belt": ["Tasks", "Health", "Finances"]
```

Arrays are invisible to locale parity tests (the test helpers skip arrays), cannot be properly validated, and create structural ambiguity.

Good:
```json
"belt": {
  "item0": "Tasks",
  "item1": "Health",
  "item2": "Finances"
}
```

To iterate flattened objects in components, use a helper:
```ts
export function objectValues<T>(raw: unknown): T[] {
  if (typeof raw !== "object" || raw === null || Array.isArray(raw)) {
    return [];
  }
  return Object.values(raw as Record<string, T>);
}
```

Usage:
```ts
const items = objectValues<string>(t("belt", { returnObjects: true }));
items.map((i) => ...);
```

## Locale Parity Tests

Every project with i18n MUST have a locale parity test (`tests/locales.test.ts`) that validates:

1. **Key parity**: Every locale has exactly the same keys as `en.json`, no missing, no extra. The `collectKeys` helper MUST include intermediate object keys (not just leaf keys) so structural differences and empty object sections are caught. Empty objects (`"section": {}`) MUST fail as extra keys.
2. **No empty values or objects**: No locale may have empty string values (`""`) or empty objects (`{}`). A `collectEmpties` helper should walk the tree and reject both.
3. **Interpolation param matching**: For every leaf key, the set of `{{param}}` tokens MUST match `en.json` exactly. A locale cannot add, remove, or rename interpolation params.
4. **Dead key detection** (recommended): Scan source code for `t("...")` calls and assert that every `en.json` key is used and every `t()` call references an existing key.

### Key test helpers

```ts
// Includes intermediate keys, not just leaf keys
function collectKeys(obj: unknown, prefix = ""): Set<string> {
  const keys = new Set<string>();
  if (!isTree(obj)) return keys;
  for (const [key, value] of Object.entries(obj)) {
    const fullKey = prefix ? `${prefix}.${key}` : key;
    keys.add(fullKey);
    if (isTree(value)) {
      for (const k of collectKeys(value, fullKey)) keys.add(k);
    }
  }
  return keys;
}

// Rejects empty strings AND empty objects
function collectEmpties(obj: unknown, prefix = ""): string[] {
  const empties: string[] = [];
  if (!isTree(obj)) return empties;
  for (const [key, value] of Object.entries(obj)) {
    const fullKey = prefix ? `${prefix}.${key}` : key;
    if (isTree(value)) {
      if (Object.keys(value).length === 0) {
        empties.push(fullKey);
      } else {
        empties.push(...collectEmpties(value, fullKey));
      }
    } else if (typeof value === "string" && value === "") {
      empties.push(fullKey);
    }
  }
  return empties;
}
```

## Currency and Pricing

Keep prices in a single base currency (e.g. USD) unless integrated with a payment provider (e.g. Adapty) that can localize prices per region. Localizing prices manually creates maintenance burden and drift. The surrounding text (e.g. "Billed at", "per year") MUST still be i18n'd, with only the numeric value passed as an interpolation param.
